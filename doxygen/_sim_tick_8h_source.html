100/** This class define the interface to provide a simulates system tick (in
101 milliseconds) to the application. The goal is provide a mechanism for
102 an application to run in 'simulate time' instead of real time. The
103 simulated tick mechanism is for the most part transparent an application,
104 i.e. the application programmer does NOT have to be aware or design
105 his code differently to be able to run in simulate tick mode. The
106 'intrusions points' are detailed below. The typical uses cases for using
107 simulated time are:
108
109@htmlonly
110<pre>
111
112 o Testing. A simulated system tick allows the test harness to have
113 deterministic control of the application's time. In addition, time
114 can be singled stepped, paused, accelerated (i.e. run N days in
115 M minutes).
116
117 o Model in the Loop (MIL). An external application such as Matlab/
118 Simulink can be used to provide the simulated tick input as well
119 as capture the application's output and provide the application
120 inputs, i.e. connecting the application to a simulated physical
121 world. Note: 'connecting' a MIL application to a simulation such
122 as Matlib/Simulink requires a communication layer that is NOT
123 provided by KIT as well as specialization to Matlib/Simulink
124 that mates up with the afore mentioned communication layer. This
125 is left as an exercise to the reader :).
126
127
128 NOTES:
129 ------
130 o All timing sources, e.g. Kit::System::sleep(), provided by KIT
131 have support for simulated ticks build in.
132
133 o All of the KIT's ITC message queues/mailbox servers, etc. also
134 have support for simulated ticks build in.
135
136 o With respect to threads that are using simulated time, There needs to
137 be at least one thread (at any given time) that is a 'timing' source,
138 e.g. waiting on sleep(), timedWait(), etc. If your application uses
139 a Kit::System::EventLoop (or a sub-class) instance - then this
140 requirement has been met. Worse case you a create a thread that
141 forever loops on sleep().
142
143 o This interface is independent of the underlying platform, i.e works
144 on ALL platforms.
145
146 o NOT all threads and/or parts of an application need or should use
147 simulated time. For example the test harness (or communication
148 interface to a simulation tool) that provide the stimulus to
149 advance the simulated tick need to run in real-time. It is the
150 Application responsibility to properly selected what is and is
151 not using simulated time.
152
153 - The platform's native/main thread ALWAYS runs in 'real-time', i.e.
154 call to SimTick::wait() never blocks. Also any sleep() calls from
155 the native/main thread are redirected to the Api::sleepInRealTime().
156
157 o THREADING: This interface provides ONLY a functional simulation of
158 system ticks. These means the actual runtime timing of threads and/or
159 sequence of execution will NOT be the same as the 'live'
160 non-simulated application.
161
162 - When using simulated ticks with multi-threaded applications, the
163 timing of the threads using the simulated ticks behavior is 'better'
164 if the thread supplying the ticks has LOWER priority than the
165 threads using the simulate ticks.
166
167 - BEWARE of multi-core targets. Because many threads can actually
168 execute at the SAME time on multi-core boxes, the timing between
169 the tick source thread and the simulated-tick threads and between
170 simulated-tick threads is UNPREDICTABLE. The tick sequencing
171 is guaranteed, but the order within a tick OR timing relations to
172 the tick source OR inter-tick timing is NOT guaranteed.
173
174 - Not all use cases using a simulated tick and threading will work
175 and/or are supported. The simulated tick logic only works for
176 event driven and/or frequency based where all actions for a time
177 period as completed in on pass of the "main loop". Other issues:
178
179 o The assumption is that all calls to a mutex will NOT cause the
180 thread to block across simulated ticks, i.e. mutex can ALWAYS
181 be successfully acquired and release within the context of
182 single simulated tick
183
184 - Be careful about deleting threads when using Simulated Ticks -
185 brute force terminating threads in this scenario has unpredictable
186 and/or improper behavior. Always have threads run to completion
187 when using Simulate Ticks.
188
189 o The DEFAULT implementation of this interface assumes that one
190 system tick EQUALS one millisecond.
191
192
193 HOW TO ADD SIMULATED TICK SUPPORT
194 ---------------------------------
195 o Enabling Simulated time is done at compile done. The following steps
196 are required:
197 1) Compile with the preprocessor symbol USE_KIT_SYSTEM_SIM_TICK
198 defined.
199
200 2) Compile & Link the src/Kit/System/_simtick directory. Also do NOT
201 compile/link the src/Kit/System/xxx/_realtime directories
202
203 o Typically thats all that needs to be done. The only additional work is if
204 application creates thread(s) that have an application specific (i.e. not
205 a KIT Event Loop, mailbox server, etc.) 'forever' loop AND it is desired
206 for the thread(s) to use 'simulated time'. For this scenario the
207 application must add a call to this interface at the top of the forever
208 loop. See the example below:
209
210 void myThreadEntryFunction( void* arg )
211 {
212 ...
213
214 while(1)
215 {
216 // Block the main loop until the next simulated system tick
217 KIT_SYSTEM_SIM_TICK_TOP_LEVEL_WAIT();
218 ...
219 }
220 }
221 }
222
223</pre>
224@endhtmlonly
225 */
226class SimTick : public Kit::Container::ListItem
227{
228public:
229 /** This method will cause the current thread to block UNTIL one
230 simulated system tick has elapsed. Every call to this method
231 consumes one simulated system tick. This method SHOULD only be used
232 at the very most top level of a thread's 'forever' loop.
233
234 This method SHOULD NEVER be called directly, but via the preprocessor
235 macro KIT_SYSTEM_SIM_TICK_TOP_LEVEL_WAIT(). This allows the simulate
236 tick code to be compiled out of production versions of the application.
237 */
238 static void topLevelWait() noexcept;
239
240 /** This method returns true if the current thread is using 'simulated time'
241
242 This method SHOULD NEVER called directly, but via the preprocessor
243 macro KIT_SYSTEM_SIM_TICK_USING_SIM_TICKS(). This allows the simulate tick
244 code to be compiled out of production versions of the application.
245 */
246 static bool usingSimTicks() noexcept;
247
248 /** This method is used to support blocking calls (such as wait on semaphore,
249 sleep(), etc.) within an individual iteration of thread's top level
250 loop. The 'block' in this case IS NOT waiting on the simulated tick -->
251 it is waiting on the application logic.
252
253 This method SHOULD NEVER called directly, but via the preprocessor
254 macro KIT_SYSTEM_SIM_TICK_APPLICATION_WAIT(). This allows the simulate
255 tick code to be compiled out of production versions of the application.
256 */
257 static void applicationWait() noexcept;
258
259 /** This method returns the current simulated tick count
260 */
261 static uint64_t current() noexcept;
262
263
264public:
265 /** Calling this method advances the application's time by 'numTicks'. This
266 method should ONLY called by a test harness or some other module that
267 is typically not part of the application, i.e. the caller of this
268 method must be 'executing' in real time. Internally the simulated
269 time is advance at one tick at time.
270
271 The method returns true after completing advancing the internal tick
272 count by the amount specified. False is only returned when the
273 simulate tick engine has detected that there a NO simulated-tick-threads
274 executing OR that ALL simulated-tick-threads are dead locked.
275 */
276 static bool advance( uint64_t numTicks ) noexcept;
277
278
279 /** This method returns true if the specified thread (by Thread ID) is
280 blocked waiting on the next simulate tick. While this method is
281 thread safe -->it is intended (and only make sense) to be called
282 from the SAME thread that also calls the advance() method.
283 */
284 static bool isWaitingOnNextTick( KitSystemThreadID_T threadID ) noexcept;
285
286
287public:
288 /** This COMPONENT Scoped method is used during thread creation to insert the
289 necessary hooks (per thread) for the simulate tick engine. If
290 'useSimTicks' is false then thread does NOT use simulated time. This
291 method SHOULD NEVER be called by the application.
292 */
293 static void threadInit_( bool useSimTicks = true ) noexcept;
294
295 /** This COMPONENT Scoped method is used during thread deletion to insert the

297 method SHOULD NEVER be called by the application.
298 */
299 static void onThreadExit_() noexcept;

301
