28/** This mostly abstract class defines the interface for a Model Point. A
29 Model Point contains an instance of a Point's data and is responsible for
30 managing thread safe access to the 'Point'. The intent of a Model point is
31 that it is the Application's canonical source for a 'Point'
32
33 NOTES:
34 1) The concrete leaf classes are responsible for ensuring type
35 safety.
36 2) All methods are atomic, i.e. a protected by the Model Base's mutex.
37 This means that all methods will block if there is a call in-progress
38 to ANY Model Point in a given Model Data Base.
39 3) All methods in this class ARE thread Safe unless explicitly
40 documented otherwise.
41 4) When a Model Point transitions (or initialized in) to the invalid set,
42 its 'data' is set to all zero so that transition from the invalid to the
43 valid state when using read-modify-write operation is consistent in
44 its behavior.
45 */

54 eUNLOCK, //!< Request to unlock the MP. If the MP is already unlocked - the request is ignored and the update operation is completed
55 };

313 /** This method is used to attach a observer to a Model Point. Once
314 attached the observer will receive a change notification (aka a
315 callback) every time the Model Point's data/state changes. Once, a
316 observer is attached - it will stay attached to the application
317 calls detach().
318
319 There is no limit to the number of observers that can attach to
320 a Model Point.
321
322 The attach() method can be called even if the observer is already
323 attached. When this happens, the attach process is 'restarted', i.e.
324 the 'initialSeqNumber' is used for the observer's sequence number.
325
326 The change-detect mechanism uses a sequence number. Each Model Point
327 and each observer has sequence number. When the observer's sequence
328 number does not equals the Model Point's sequence number - the observer
329 receives a change notification and the observer's sequence number is
330 updated to match the Model Point's sequence number at the time of
331 change notification. When a observer attaches to Model Point with
332 an the 'seqNumber' argument set to SEQUENCE_NUMBER_UNKNOWN, the
333 observer will get an 'immediate' change notification.
334
335 The callbacks for the Change Notifications are called as part of the
336 Event Queue server. As part of the asynchronous processing (timers,
337 ITC, EventFlags, etc.) of the Event Queue server will also process
338 all pending Change Notification and invoke the call backs. What does
339 that all mean? The Change notifications are "local" to the observer's
340 thread very similar to how the Kit::Timers work. It also means that
341 no change notification callback will be called till the Event Queue server
342 loops back to the "top" of its forever loop.
343
344 NOTE: The Change Notification mechanism does NOT guarantee that a
345 client will receive a notification for EVERY change to a Model
346 Point (i.e. 'fast edges' could/will be missed). What is guaranteed
347 is that once a Model Point's value has 'settle' all registered
348 observers will have received change notification for the Model
349 Point's current value.
350 */
351 virtual void attachObserver( IObserver& observer,
352 uint16_t initialSeqNumber = SEQUENCE_NUMBER_UNKNOWN ) noexcept = 0;
353
354 /** This method is used to detach an observer from a Model Point. See the
355 attach() method for more details about the Subscription/Change
356 Notification mechanism.
357
358 The detach() method can be called even if the observer is NOT
359 currently attached. The detach() method can be called within the
360 Change Notification callback.
361 */
362 virtual void detachObserver( IObserver& observer ) noexcept = 0;
