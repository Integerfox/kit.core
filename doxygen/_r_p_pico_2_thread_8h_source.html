33/** This concrete class implements a Thread object that maps 1 to 1 with
34 a processor core on the Raspberry PI RP2xxx micro-controller.
35
36 At least one thread must be created and at most only two threads can be
37 created. The first thread created executes on core0. The second thread
38 created executes on core1.
39
40 The first thread MUST be created before enableScheduling() is called. The
41 second thread that executes on core1 can be created or after enableScheduling()
42 is called. The core1 thread can also be destroyed and recreated after
43 enableScheduling() is called.
44
45 The name of the threads are hard coded to 'CORE0' and 'CORE1'.
46
47 The 'native' thread handle is the core number (i.e. 0 or 1).
48
49 Threads can ONLY be created using the Kit::System::Thread::create() method.
50 When created threads:
51 - Only the 'runnable' and 'name' should be specified.
52 - The stack sizes are defined by the build symbols: PICO_STACK_SIZE
53 and PICO_CORE1_STACK_SIZE, e.g. -DPICO_STACK_SIZE=4096 -DPICO_CORE1_STACK_SIZE=2048
54 - Thread priority has NO meaning since both threads/cores execute concurrently.
55 - The thread(s) begin executing after enableScheduling() is called.
56
57 NOTES:
58 1. The threads do NOT 'execute' (i.e. call their Runnable object's run()
59 method till enableScheduling() is called. In addition the enableScheduling()
60 method NEVER returns.
61 2. The suspendScheduling() semantics suspends ALL processing on the 'other'
62 core. This includes interrupt processing on the other core.
63
64 */
