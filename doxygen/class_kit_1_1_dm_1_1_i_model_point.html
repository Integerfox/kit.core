Detailed Description

This mostly abstract class defines the interface for a Model Point.

A Model Point contains an instance of a Point's data and is responsible for managing thread safe access to the 'Point'. The intent of a Model point is that it is the Application's canonical source for a 'Point'

NOTES: 1) The concrete leaf classes are responsible for ensuring type safety. 2) All methods are atomic, i.e. a protected by the Model Base's mutex. This means that all methods will block if there is a call in-progress to ANY Model Point in a given Model Data Base. 3) All methods in this class ARE thread Safe unless explicitly documented otherwise. 4) When a Model Point transitions (or initialized in) to the invalid set, its 'data' is set to all zero so that transition from the invalid to the valid state when using read-modify-write operation is consistent in its behavior.

◆ attachObserver()

virtual void Kit::Dm::IModelPoint::attachObserver ( IObserver & observer,
uint16_t initialSeqNumber = SEQUENCE_NUMBER_UNKNOWN )
protectedpure virtualnoexcept

This method is used to attach a observer to a Model Point.

Once attached the observer will receive a change notification (aka a callback) every time the Model Point's data/state changes. Once, a observer is attached - it will stay attached to the application calls detach().

There is no limit to the number of observers that can attach to a Model Point.

The attach() method can be called even if the observer is already attached. When this happens, the attach process is 'restarted', i.e. the 'initialSeqNumber' is used for the observer's sequence number.

The change-detect mechanism uses a sequence number. Each Model Point and each observer has sequence number. When the observer's sequence number does not equals the Model Point's sequence number - the observer receives a change notification and the observer's sequence number is updated to match the Model Point's sequence number at the time of change notification. When a observer attaches to Model Point with an the 'seqNumber' argument set to SEQUENCE_NUMBER_UNKNOWN, the observer will get an 'immediate' change notification.

The callbacks for the Change Notifications are called as part of the Event Queue server. As part of the asynchronous processing (timers, ITC, EventFlags, etc.) of the Event Queue server will also process all pending Change Notification and invoke the call backs. What does that all mean? The Change notifications are "local" to the observer's thread very similar to how the Kit::Timers work. It also means that no change notification callback will be called till the Event Queue server loops back to the "top" of its forever loop.

NOTE: The Change Notification mechanism does NOT guarantee that a client will receive a notification for EVERY change to a Model Point (i.e. 'fast edges' could/will be missed). What is guaranteed is that once a Model Point's value has 'settle' all registered observers will have received change notification for the Model Point's current value.

◆ setInvalid()

virtual uint16_t Kit::Dm::IModelPoint::setInvalid ( bool forceChgNotification = false,
LockRequest_T lockRequest = eNO_REQUEST )
pure virtualnoexcept

This method is used to mark the element's data as invalid.

The method returns the Model Point's sequence number after updating the valid state.

Notes: 1) Any write operation will set the Model Point's state to valid. 2) Change Notification(s) will ONLY be triggered when: a) forceChgNotification is true, OR b) there is valid to invalid state transition (i.e. if the Model Point is already in the invalid state - no change notification will be triggered), OR 3) If the Model Point is locked then the invalidate operation... a) If lockRequest == eNO_REQUEST, the operation silently fails, i.e. nothing is done. OR b) If lockRequest != eNO_REQUEST, the operation is performed and the the new lock state is applied

This updates the lock state of the Model Point.

Model Points support the concept of a client 'locking' the MP's data value. When a MP's data has been locked - any attempted writes/updated operation to the MP will SILENTLY fail. The Application uses the 'eUNLOCK' lock request to remove the locked state from the MP's data.

This method never triggers change notification(s).

This method converts the Model Point's data to JSON string and copies the resultant string into 'dst'.

If the Model Point's data cannot be represented as a JSON object then the contents of 'dst' is set to an empty string and the method returns false; else the method returns true. The format of the string is specific to the concrete leaf class. However, it is strongly recommended that the output of this method be the same format that is expected for the fromJSON() method.

The general output format:

{ name:"<mpname>", type:"<mptypestring>", valid:true|false, seqnum:nnnn, locked:true|false, val:<value> }
Notes:

- The 'val' key/value pair is omitted if the MP is in the invalid state
- The 'val' key/value pair can be a single element, an object, or
